# DFT IDE

[中文文档](README.zh-CN.md)

DFT IDE is a VS Code extension that turns VS Code into a DFT workflow console. It provides a project home page, DFT flow navigation, common configuration, design flow, verification flow, project creation, VS Code integration demos, and reserved backend API integration points.

The current codebase is a demo foundation. It is intended to be extended into a project-oriented IDE where different projects load different COMMON, Design, Verification, Formal, and STA configurations.

## Tech Stack

- VS Code Extension API: extension activation, commands, Activity Bar view container, Tree View, Webview Panel, notifications, Quick Pick, terminal, settings, file opening, and workspace APIs.
- TypeScript: shared language for the extension host and webview UI.
- React 19: webview application framework.
- Ant Design 5: UI component library for forms, cards, lists, buttons, layout, and workflow pages.
- Zustand: lightweight webview state management for active project and flow context.
- react-hook-form + zod + @hookform/resolvers: form state and validation examples.
- esbuild: bundles extension host code and browser webview code.
- npm-run-all: runs extension and webview builds in parallel.

Installed but not heavily used yet:

- @tanstack/react-query: reserved for server state, project/config fetching, job polling, and cache invalidation.
- @tanstack/react-table: reserved for richer tables such as reports, pin tables, coverage views, and job history.

## Development

Install dependencies:

```bash
npm install
```

Build once:

```bash
npm run compile
```

Watch both extension and webview bundles:

```bash
npm run watch
```

Type-check:

```bash
npm run check
```

Run in VS Code:

1. Open this repository in VS Code.
2. Run `npm run compile` once, or keep `npm run watch` running.
3. Press `F5` to launch the Extension Development Host.
4. Use the DFT IDE Activity Bar / side view to open Home, COMMON, Design, or Verification.

Note: the Extension Development Host does not always behave exactly like a real installed extension. Window and webview restoration are more reliable after packaging and installing the extension normally.

## Build Outputs

The extension uses two esbuild bundles:

- `out/extension.js`: CommonJS bundle for the VS Code extension host.
- `out/webview.js`: browser bundle for the React webview.

`vscode:prepublish` runs `npm run compile`, so packaging should build both bundles.

Package as VSIX:

```bash
npx @vscode/vsce package
```

The `.vscodeignore` file excludes source files, local editor config, logs, maps, and development-only files from the VSIX. The packaged extension should contain `package.json`, `README.md`, `CHANGELOG.md`, assets, and the compiled `out/` bundle.

## VS Code Extension Host

Main file: `src/extension.ts`

Responsibilities:

- Registers the DFT IDE Activity Bar container and Tree View.
- Registers commands:
  - `dftIde.openWelcome`
  - `dftIde.openFlow`
  - `dftIde.createProject`
  - `dftIde.applyLayout`
  - `dftIde.restoreLayout`
- Creates and manages the webview panel.
- Handles webview IPC messages:
  - path selection
  - opening files/folders
  - project creation
  - mock job submission/status polling
  - VS Code capability demos
- Applies optional DFT-focused workbench layout settings.

## Webview App

Main files:

- `src/webview/main.tsx`
- `src/webview/App.tsx`
- `src/webview/components/Welcome.tsx`

The webview is a React app embedded in a VS Code Webview Panel. It receives initial state from the extension host through injected globals:

- `window.DFT_IDE_API_BASE`
- `window.DFT_IDE_INITIAL_VIEW`

Runtime messages are passed through `vscode.postMessage` and handled by the extension host.

## Project API Integration

The current project list uses a mock fallback in:

```text
src/webview/services/projectService.ts
```

When `dftIde.apiBase` is empty, the home page uses local mock projects. When configured, the webview calls backend endpoints:

```text
GET  {apiBase}/api/dft-ide/projects/dashboard
POST {apiBase}/api/dft-ide/projects/{projectId}/select
```

Expected dashboard response:

```ts
interface ProjectDashboard {
  projects: DftProject[];
  currentProjectId: string | null;
}

interface DftProject {
  id: string;
  name: string;
  rootPath: string;
  owner: string;
  updatedAt: string;
  stage: string;
  description: string;
}
```

The VS Code setting is declared in `package.json`:

```json
"dftIde.apiBase": ""
```

## State Model

Main store:

```text
src/webview/store/wizardStore.ts
```

Important state:

- `activeProject`: selected project identity and root path.
- `flowContext`: current flow category and optional project id.
- `taskPayload`: shared payload used by the older wizard flow.

When project-specific configuration is added, prefer loading it from the selected `activeProject` / `flowContext.projectId` rather than hard-coding values inside individual step components.

## Adding a New Flow

Typical steps:

1. Add the flow item to `FLOW_CONFIGS` in `src/extension.ts`.
2. Add metadata in `flowMeta` in `src/webview/App.tsx`.
3. Add a new flow component under `src/webview/flows/`.
4. Reuse `src/webview/components/shared/FlowShell.tsx` for step-based flows.
5. Add the flow to `renderFlowContent` in `App.tsx`.
6. Add step components under `src/webview/components/<flow-name>/`.

For example, Formal and STA already exist as planned entries in the UI and can follow the Design / Verification pattern.

## IPC Pattern

Webview-to-extension helpers live in:

```text
src/webview/utils/ipc.ts
```

Use `ipcRequest` style calls when a response is needed, such as selecting a path. Use direct `vscode.postMessage` for one-way actions, such as opening a file or running a VS Code capability demo.

Extension-side message handling is in `currentPanel.webview.onDidReceiveMessage` inside `src/extension.ts`.

## VS Code Layout Mode

The extension exposes a DFT-focused layout mode through settings:

```json
"dftIde.layout.autoApply": true,
"dftIde.layout.hideMenuBar": true,
"dftIde.layout.hideActivityBar": true
```

Commands:

- `DFT IDE: Apply Layout`
- `DFT IDE: Restore VS Code Layout`

The layout code backs up existing global settings before applying changes, then restores them when requested.

## Coding Notes

- Keep extension host code VS Code API-oriented and avoid browser-only APIs there.
- Keep webview code browser-safe and communicate with VS Code through IPC.
- Add direct dependencies to `package.json` when source code imports them directly.
- Prefer shared components in `components/shared` for layout and reusable controls.
- Keep project-specific data behind service functions so backend integration can replace mock data cleanly.
- Run both `npm run compile` and `npx tsc --noEmit` before handing off larger changes.
